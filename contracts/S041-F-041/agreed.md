## 1. Goal

Add `POST /api/recipes/{id}/preview` — a dry-run endpoint that pulls up to 5
candidate chunks from Lance, calls the LLM gateway once per chunk, and returns
3–5 synthesised `{instruction, output}` samples **without writing anything to
MinIO or Postgres**.

## 4. Schemas

### 4.1 Request — `RecipePreviewRequest`

```python
class RecipePreviewRequest(BaseModel):
    model_config = ConfigDict(extra="ignore")

    n_samples: int = Field(default=3, ge=3, le=5)
```

- `n_samples`: number of preview samples to generate; default 3; Pydantic enforces [3, 5]; a value outside that range returns **422** automatically (FastAPI validation, no custom handler needed).

### 4.2 Response — `RecipePreviewResponse`

```python
class RecipePreviewResponse(BaseModel):
    samples: list[dict[str, Any]]
```

- `samples`: list of template-output dicts. For `sft_synthesis_qa` each item has at minimum `"instruction"` (str) and `"output"` (str). The dict type is intentionally loose at the API layer because future templates will emit different shapes; a discriminated union can be added in a later sprint without breaking this contract.
- There is **no `SamplesItem` wrapper** in MVP — the shape is template-dependent and a per-template typed union is out of scope.

### 4.3 OpenAPI shape (informational)

```
POST /api/recipes/{id}/preview
  requestBody:  RecipePreviewRequest   { n_samples?: integer [3..5] }
  response 200: RecipePreviewResponse  { samples: array[object]     }
```

## 5. Handler Outline

Location: `apps/api/dataplat_api/routers/recipes.py`, registered as:

```python
@router.post("/{id}/preview", response_model=RecipePreviewResponse)
async def preview_recipe(
    id: int,
    body: RecipePreviewRequest,
    current_user: User = Depends(get_current_user),
    session: AsyncSession = Depends(get_session),
    llm: LLMGateway = Depends(get_llm_gateway),
) -> RecipePreviewResponse:
```

**Steps:**

1. **Owner-scoped load** — execute `select(Recipe).where(Recipe.id == id).where(Recipe.owner_id == current_user.id)`. If `scalar_one_or_none()` is `None`, raise `HTTP 404 "Recipe not found"`. (Mirrors `get_recipe` exactly — collapses not-found and wrong-owner into a single 404, no enumeration leak.)

2. **Template validation** — navigate `recipe.definition` with `.get()` to extract the schema section and its nested config:
   ```python
   schema_section = definition.get("schema", {})   # e.g. {"template": "sft_synthesis_qa", "config": {...}}
   template       = schema_section.get("template")
   config         = schema_section.get("config", {})  # inner dict passed to the per-template generator
   ```
   - If `template` is `None` or absent → `HTTP 400 "Recipe definition missing required field: schema.template"`.
   - If `template` is not in the dispatch table (i.e. not `"sft_synthesis_qa"` for MVP) → `HTTP 400 "Preview supports only 'sft_synthesis_qa' in MVP; got '{template}'"`.
   
   The dispatch table check is a single `if template not in _TEMPLATE_HANDLERS` test (see §6).
   
   `config` (the inner `definition["schema"]["config"]` subsection) is what gets passed to `run_preview` and subsequently to the per-template generator. The generator never receives the full `schema_section` — it sees only `config`.

3. **Extract filter** — `where_clause: str | None = definition.get("filter", {}).get("where")`. May be `None` (no filter applied to Lance query).

4. **Query Lance for candidate chunks** — call `await run_preview(where_clause, body.n_samples, template, config, llm)` (see §6 for full signature). `run_preview` wraps Lance I/O in `asyncio.to_thread`. On Lance I/O error it raises `LanceQueryError`; the router catches `LanceQueryError` and converts to `HTTP 400 "Lance query error: {exc}"`. If `run_preview` returns an empty list it raises `PreviewError(status_code=400, detail="No matching chunks for preview; check recipe filter.where")` directly (see OQ-1 decision).

5. **Fan-out LLM calls** — inside `_generate_samples_sft_synthesis_qa` (called by `run_preview`), `await asyncio.gather(*[_generate_sft_qa(chunk, config, llm) for chunk in chunks])`. `_generate_sft_qa` is the private per-chunk async helper; `_generate_samples_sft_synthesis_qa` is the dispatch-table entry that owns the gather loop. Each per-chunk call renders the prompt and calls `llm.complete(messages, max_tokens=512)`. The gather ensures all `n_samples` calls happen concurrently, keeping total latency close to a single LLM round-trip rather than n × RTT (see OQ-3 decision).

6. **Assemble and return** — `return RecipePreviewResponse(samples=samples)`.

**No writes** to MinIO, Parquet, or any Postgres table are performed at any step. All intermediate data (chunks list, LLM responses) lives only in process memory (invariant #2).

## 6. Schema-Template Dispatch Design

### 6.0 `PreviewError` exception and `run_preview` entry-point

**`PreviewError`** — a new exception class defined at the top of `dataplat_api/recipes/preview.py`:

```python
class PreviewError(Exception):
    """Raised by preview helpers when a request-level error should become an HTTPException."""

    def __init__(self, status_code: int, detail: str) -> None:
        super().__init__(detail)
        self.status_code = status_code
        self.detail = detail
```

The router catches `PreviewError` once and converts it to `HTTPException`:

```python
try:
    samples = await run_preview(where_clause, body.n_samples, template, config, llm)
except PreviewError as exc:
    raise HTTPException(status_code=exc.status_code, detail=exc.detail)
```

**`run_preview` signature:**

```python
async def run_preview(
    where_clause: str | None,
    n_samples: int,
    template: str,
    config: dict,
    llm: LLMGateway,
) -> list[dict[str, Any]]:
```

- Runs Lance I/O in `asyncio.to_thread`; re-raises `LanceQueryError` on Lance failures (router already catches `LanceQueryError` → HTTP 400, consistent with `chunks.py` precedent).
- If Lance returns zero rows: raises `PreviewError(status_code=400, detail="No matching chunks for preview; check recipe filter.where")`.
- Dispatches to the appropriate handler from `_TEMPLATE_HANDLERS[template]`, passing `(chunks, config, llm)` — where `config` is the inner `definition["schema"]["config"]` subsection (NOT the full `schema_section`).
- Does NOT raise `HTTPException` for LLM failures — those are raised as `PreviewError` by the per-chunk coroutine and propagate through `asyncio.gather` to the outer `try/except`.

**Lance column projection** — fetch exactly the columns a `prompt_template` may legitimately reference:

```python
_PREVIEW_COLUMNS: list[str] = [
    "chunk_id", "source_id", "text", "token_count",
    "source_refs", "attr_quality_score", "attr_lang_code",
]
```

Any `{field_name}` in a custom `prompt_template` that is not present in the returned chunk dict raises a `KeyError` during `str.format`, which the per-chunk helper catches and converts to `PreviewError(400, "Prompt template references unknown chunk field: '{field_name}'")`.

**Fallback truncation constant:**

```python
_FALLBACK_INSTRUCTION_MAX_CHARS: int = 200  # truncate chunk text for fallback instruction field
```

### 6.1 Dispatch table (in `apps/api/dataplat_api/recipes/preview.py`)

```python
# Type alias for a template handler coroutine:
#   (chunks: list[dict], config: dict [= definition["schema"]["config"]], gateway: LLMGateway)
#   -> Awaitable[list[dict]]
TemplateHandler = Callable[
    [list[dict], dict, LLMGateway],
    Awaitable[list[dict]],
]

_TEMPLATE_HANDLERS: dict[str, TemplateHandler] = {
    "sft_synthesis_qa": _generate_samples_sft_synthesis_qa,
}
```

- The dispatch call inside `run_preview`:
  ```python
  handler = _TEMPLATE_HANDLERS[template]
  return await handler(chunks, config, llm)  # config is schema.config subsection — NOT schema_section
  ```
- Adding a new template in a future sprint requires only:
  1. Writing the new `async def _generate_samples_<template>(chunks, config, llm)` function.
  2. Registering it in `_TEMPLATE_HANDLERS`.
  3. No changes to the router, schemas, or dispatch logic.

### 6.2 `sft_synthesis_qa` handler and per-chunk helper (MVP implementation)

`_generate_samples_sft_synthesis_qa` is the registered handler; it owns the gather loop and delegates each chunk to the private per-chunk helper `_generate_sft_qa`:

```python
async def _generate_samples_sft_synthesis_qa(
    chunks: list[dict],
    config: dict,       # definition["schema"]["config"] subsection
    llm: LLMGateway,
) -> list[dict]:
    return list(await asyncio.gather(*[_generate_sft_qa(chunk, config, llm) for chunk in chunks]))
```

The per-chunk helper:

```python
async def _generate_sft_qa(
    chunk: dict,
    config: dict,
    llm: LLMGateway,
) -> dict:
```

The name `_generate_sft_qa` (referenced in §5 Step 5) refers to this private per-chunk function. It is **not** the same as `_generate_samples_sft_synthesis_qa` (which takes all chunks and manages the gather).

**Per-chunk logic inside `_generate_sft_qa`:**

1. **Render prompt** — `prompt_template: str = config.get("prompt_template", _DEFAULT_PROMPT_TEMPLATE)`. Render via `prompt_template.format(**chunk)`. If `str.format` raises `KeyError` for a field name `fn`, catch it and raise `PreviewError(status_code=400, detail=f"Prompt template references unknown chunk field: '{fn}'")`.

   `_DEFAULT_PROMPT_TEMPLATE` (module-level constant) references only the guaranteed `text` field:
   ```
   "Generate an instruction-response pair from the following text.\n"
   "Respond with valid JSON: {\"instruction\": \"...\", \"output\": \"...\"}.\n\n"
   "Text:\n{text}"
   ```

2. **LLM call** — `response = await llm.complete(messages=[{"role": "user", "content": rendered}], max_tokens=512)`.

3. **Parse response** — attempt `json.loads(response.content)`. On `json.JSONDecodeError`:
   - If `config.get("fallback_on_failure", False)` is truthy: produce `{"instruction": chunk.get("text", "")[:_FALLBACK_INSTRUCTION_MAX_CHARS], "output": response.content}`.
   - Otherwise: raise `PreviewError(status_code=status.HTTP_502_BAD_GATEWAY, detail=f"LLM returned non-JSON output and fallback_on_failure is false")`.

4. **Validate parsed dict** — check that `parsed` is a `dict` with both `"instruction"` and `"output"` keys; if not, apply the same fallback/`PreviewError(502)` logic as step 3 (non-JSON-object response treated identically to JSON parse failure).

5. **Return** the parsed sample dict.

## 7. Error Matrix

| HTTP | Trigger | `detail` string |
|------|---------|----------------|
| **200** | All steps succeed | `{"samples": [...]}` |
| **400** | `schema.template` key absent from `recipe.definition` | `"Recipe definition missing required field: schema.template"` |
| **400** | `schema.template` is not `"sft_synthesis_qa"` (or any future registered key) | `"Preview supports only 'sft_synthesis_qa' in MVP; got '{template}'"` |
| **400** | Lance/DataFusion rejects the `filter.where` expression or any other Lance I/O error | `"Lance query error: {exc}"` |
| **400** | Lance returns zero rows matching the filter | `"No matching chunks for preview; check recipe filter.where"` |
| **400** | `prompt_template` references a field not present in the chunk dict | `"Prompt template references unknown chunk field: '{field_name}'"` |
| **401** | Missing or invalid bearer token | (handled by `get_current_user` — standard 401) |
| **404** | Recipe id not found, or recipe owned by another user | `"Recipe not found"` |
| **422** | `n_samples` < 3 or > 5 (Pydantic validation failure) | (FastAPI standard 422 body) |
| **502** | LLM returns non-JSON and `fallback_on_failure` is false/absent | `"LLM returned non-JSON output and fallback_on_failure is false"` |

## 8. Open Questions — Recommendations

### OQ-1: Empty Lance result → 200 `{samples:[]}` or 400?

**Recommendation: 400.**

The verification spec requires `{"samples": [<3-5 items>]}`. Returning `{samples:[]}` would silently pass V1 as "technically valid JSON" but fail the count assertion. More importantly, zero-sample preview gives the caller no useful information — the most helpful signal is to tell them their filter matches nothing. `HTTP 400 "No matching chunks for preview; check recipe filter.where"` is actionable.

### OQ-2: Prompt template substitution — `str.format` or Jinja2?

**Recommendation: `str.format` for MVP.**

`str.format` is zero-dependency and sufficient for the simple `{text}`, `{source_id}` style substitutions expected from the recipe DSL. A Jinja2 sandbox introduces an additional dependency and escaping complexity. If template authors need conditionals or loops, Jinja2 can be swapped in for a single-function change with no API contract impact. Document the `str.format` constraint in the module docstring.

### OQ-3: Parallel vs. sequential LLM calls?

**Recommendation: `asyncio.gather` (parallel).**

With `n_samples` capped at 5 and a real Anthropic latency of ~2–6 s per call, sequential calls would routinely exceed 30 s. `asyncio.gather` keeps total latency close to a single call's RTT. The gateway's `AsyncAnthropic` client is already thread-safe for concurrent calls (it manages its own connection pool). The 30-second budget is much easier to meet with parallel calls. If real-world testing reveals rate-limit issues, a `asyncio.Semaphore(3)` guard can be added without any contract change.

### OQ-4: Non-JSON LLM response with `fallback_on_failure=false` → 502 or 500?

**Recommendation: 502 Bad Gateway.**

The cause of the failure is the upstream LLM returning content that this service cannot process. `502` is semantically correct: our service is acting as a gateway to an upstream (the LLM), and the upstream returned an unusable response. `500` implies a bug in our code, which is misleading. `502` also gives the caller a clear signal to inspect their `prompt_template` or retry.

### OQ-5: Preview owner-scoping — should a non-owner ever be allowed to preview a recipe?

**Recommendation: No (owner-scoped 404, same as `get_recipe` and `update_recipe`).**

Design doc §11.6 explicitly defers repository-level ACL. MVP uses `visibility = private|internal` only, and no sharing mechanism is defined for recipes. A non-owner trying to preview another user's recipe returns 404 (not 403) to avoid revealing that the recipe exists. This is identical to the pattern used by `get_recipe` and `update_recipe` and is the correct MVP posture.

## 9. Invariants Checklist

| Invariant | How this endpoint satisfies it |
|-----------|-------------------------------|
| **#1 Lineage mandatory** | N/A — preview produces no Commit, no dataset row, and no lineage records. All intermediate data is discarded after the response. |
| **#2 Storage separation + CAS — no blob bytes in Postgres, no MinIO writes** | Preview is entirely in-memory. Chunks are read from Lance into a Python `list[dict]` and immediately discarded after the LLM calls. No `PUT`/`write` to MinIO. No INSERT/UPDATE to any Postgres table. |
| **#3 Schema frozen post-publish** | N/A — preview is read-only; it never modifies a recipe definition or any published commit, so the freeze guard does not apply. |
| **#4 LLM calls go through the gateway** | All LLM calls use `llm.complete(...)` where `llm: LLMGateway = Depends(get_llm_gateway)`. The `_generate_sft_qa` per-chunk helper takes `LLMGateway` as a parameter — it never imports `anthropic` directly. |
| **#5 Async SQLAlchemy — no sync sessions, no `session.query()`** | The owner-scoped recipe load uses `await session.execute(select(Recipe).where(...))` with `AsyncSession`. Lance I/O runs in `asyncio.to_thread(_sync_lance_fn)` — the synchronous `get_or_create_chunks_table()` call never blocks the event loop directly. |
| **#6 OpenAPI ↔ TS type sync** | `RecipePreviewRequest` and `RecipePreviewResponse` are new Pydantic models added to `schemas/recipes.py`. After implementation, `make codegen` MUST be run and the updated `packages/api-types/openapi.json` committed in the **same** commit as the Python changes. |

## 10. Out of Scope (Explicit)

The following are **not** part of this sprint and MUST NOT be implemented:

| Item | Rationale |
|------|-----------|
| Template handlers for `cpt_plain`, `cpt_concat`, `sft_from_chunks`, `dpo_two_model`, `benchmark_mcq`, `benchmark_open` | Only `sft_synthesis_qa` is required for MVP (design doc §12.1). The dispatch table is designed so adding them is mechanical. |
| Real LLM latency budgeting / timeout enforcement in the handler | The 30-second requirement is a test-harness check against deterministic mocks. Production latency is governed by the Anthropic SDK's own timeout configuration, not by a `asyncio.wait_for` wrapper in the handler. |
| JSON Schema validation of `recipe.definition` structure | Explicitly deferred to F-082. The handler does only the minimal `.get()` key extraction it needs; it does not validate the full DSL shape. |
| Rate limiting / concurrency cap on preview calls | Post-MVP concern. A `Semaphore` guard on `asyncio.gather` can be added if Anthropic rate limits become a problem in practice. |
| Storing or caching preview results | Preview is ephemeral by definition. Results are never written anywhere. |
| ACL / sharing — allowing non-owners to preview | Deferred per design doc §11.6. MVP is owner-scoped only. |
| Streaming / SSE response for progressive sample delivery | Post-MVP UX improvement. The endpoint is synchronous-response only. |
