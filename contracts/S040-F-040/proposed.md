**Feature:** F-040 — Update recipe endpoint
**Goal:** Implement `PUT /api/recipes/{id}` that updates a recipe's `definition` (and optionally `description`), bumps `updated_at`, and rejects updates when any dataset has already been materialized from the recipe (schema-frozen-post-publish, invariant #3).

## 1. What Will Be Built

The handler is owner-scoped (same 404-for-both-not-found-and-wrong-owner pattern as `get_recipe`), enforces the post-publish freeze (checks `dataset` table for any row with `recipe_id == id` using an `exists()` query), applies the update, explicitly bumps `updated_at`, commits, and returns the full `RecipeOut`.

## 3. Schema Design — `RecipeUpdate`

### Decision: `description` IS updatable

**Rationale:** `description` is free-form human text, carries no schema/transformation semantics, and does not affect dataset lineage or reproducibility. Mutating it poses no risk to existing datasets. Locking it alongside `definition` would be needlessly restrictive. Invariant #3 ("schema frozen post-publish") concerns the *transformation contract* — `definition` is the contract; `description` is a label. Allowing `description` updates post-materialization is therefore safe and consistent with the spec's spirit.

`name` is NOT in `RecipeUpdate`. The spec says "updates the recipe definition"; `name` has a uniqueness constraint whose mutation semantics (rename, re-enumeration risk) are non-trivial and explicitly out of scope for MVP.

### Model definition (pseudocode)

```python
class RecipeUpdate(BaseModel):
    """Request body for PUT /api/recipes/{id} (F-040).

    ``definition`` is REQUIRED — the whole definition dict is replaced (not merged).
    Typed as dict[str, Any] to reject non-object JSON (array, null, bare string) with
    422 at the Pydantic boundary, matching RecipeCreate semantics.

    ``description`` is OPTIONAL — may be omitted to leave the existing description
    unchanged, or sent as null/None to clear it, or sent as a string to update it.
    The handler uses ``body.model_fields_set`` to distinguish "caller omitted the field"
    (leave existing value) from "caller explicitly sent null or a string" (apply it).
    This is the canonical Pydantic v2 idiom for partial-update fields — no non-serializable
    sentinel, no PydanticJsonSchemaWarning, clean ``default: null`` in the OpenAPI schema.

    ``name`` is NOT present — renaming is deferred; MVP locks names after creation.
    """
    model_config = ConfigDict(extra="ignore")

    definition: dict[str, Any]      # required; full replacement
    description: str | None = None  # optional; None means "clear it" when present in body;
                                    # absence (not in model_fields_set) means "leave unchanged"
```

**`description` field semantics via `model_fields_set`:**

| Caller sends | `"description" in body.model_fields_set` | `body.description` | Handler action |
|---|---|---|---|
| Omits `description` entirely | `False` | `None` (default) | Leave `recipe.description` unchanged |
| `"description": null` | `True` | `None` | Set `recipe.description = None` (clear) |
| `"description": "some text"` | `True` | `"some text"` | Set `recipe.description = "some text"` |

**Field constraints:**

| Field | Type | Required | Default in OpenAPI | Constraint |
|---|---|---|---|---|
| `definition` | `dict[str, Any]` | **Yes** | — | Any JSON object; non-object → 422 |
| `description` | `str \| None` | No | `null` | No length cap at API boundary (mirrors RecipeCreate) |

## 4. Handler Logic (Numbered Steps)

```
async def update_recipe(
    id: int,
    body: RecipeUpdate,
    current_user: User = Depends(get_current_user),
    session: AsyncSession = Depends(get_session),
) -> RecipeOut:
```

Note: The `@router.put("/{id}")` decorator does **not** include a `summary=` kwarg, keeping it consistent with the other three recipe handlers (F8 resolved).

1. **Auth** — `get_current_user` dependency enforces a valid JWT. No token → 401 before handler body.

2. **Load recipe (owner-scoped)** — Execute:
   ```sql
   SELECT * FROM recipe WHERE id = :id AND owner_id = :owner_id
   ```
   using `select(Recipe).where(Recipe.id == id).where(Recipe.owner_id == current_user.id)` + `scalar_one_or_none()`. If `None` → raise `HTTPException(404, detail="Recipe not found")`. This collapses "not found" and "wrong owner" into the same response — no enumeration leak (mirrors `get_recipe`).

3. **Dataset-existence check (post-publish freeze)** — Execute:
   ```sql
   SELECT EXISTS (SELECT 1 FROM dataset WHERE recipe_id = :id)
   ```
   using `select(exists().where(Dataset.recipe_id == recipe.id))` + `scalar_one()`. `exists()` short-circuits at the first matching row — correct and more efficient than `count(*)` for a binary question. If `True` → raise `HTTPException(409, detail="Recipe is locked: a dataset has been materialized from it")`.

4. **Apply patch** —
   - `recipe.definition = body.definition`  (always — it is required)
   - `if "description" in body.model_fields_set: recipe.description = body.description`
     (canonical Pydantic v2 `model_fields_set` idiom: checks whether the caller included `description` in the request body; if omitted, the existing value is untouched; see §3 for full semantics table)

5. **Bump `updated_at`** — Use app-side UTC time:
   ```python
   from datetime import datetime, timezone
   recipe.updated_at = datetime.now(tz=timezone.utc)
   ```
   See §6 for rationale (testability preference; `func.now()` is technically valid but complicates unit tests).

6. **Commit and refresh** —
   ```python
   await session.commit()
   await session.refresh(recipe)
   ```

7. **Return** — `return RecipeOut.model_validate(recipe)`

## 5. HTTP Status Code Choice: 409 vs 422

**Decision: 409 Conflict**

| Code | Semantics | Applicability here |
|---|---|---|
| 422 Unprocessable Entity | The request body is syntactically valid JSON but semantically invalid at the *input data* level (e.g. a field fails a business rule derivable from the payload alone). | Technically applicable, but 422 implies the *body* is the problem. Here the body is perfectly fine; the problem is the *server-side resource state*. |
| 409 Conflict | The request cannot be completed because it **conflicts with the current state of the resource** (RFC 9110 §15.5.10). | Precisely correct: the recipe exists, the update is well-formed, but the current state of the system (a dataset row exists) makes the mutation impossible. |

409 is semantically tighter and communicates "retry only makes sense after the conflicting state changes" — which is exactly the user's situation (they need to create a new recipe version instead). FastAPI also surfaces 409 cleanly in OpenAPI without the field-error envelope that 422 normally implies. Codebase precedent: F-037 already uses 409 for the duplicate-name conflict.

*OQ1 resolved: ACCEPTED by reviewer.*

## 6. `updated_at` Bump Strategy: App-side `datetime.now(UTC)` vs Server `func.now()`

**Decision: App-side `datetime.now(tz=timezone.utc)`**

| Approach | Pros | Cons |
|---|---|---|
| `func.now()` in UPDATE | Server clock; consistent with `created_at` `server_default=text("now()")` | Requires special unit-test handling: because `func.now()` renders as a SQL expression server-side, the Python object has no concrete `updated_at` value before a DB round-trip. Mocking this in tests that don't hit a real DB is more involved. |
| `datetime.now(tz=timezone.utc)` | Works naturally with SQLAlchemy ORM attribute assignment; timezone-aware; trivially mockable in tests; value is available on the Python object immediately | Slight clock skew possible if app and DB are on different hosts — negligible for MVP |

**Accurate rationale for choosing app-side UTC:** We prefer `datetime.now(tz=timezone.utc)` because it is trivially mockable in tests and round-trips through the ORM as a concrete Python value. `func.now()` is technically valid in SQLAlchemy 2.0 — the ORM will render it in the `UPDATE SET` clause as a server-side SQL call (`UPDATE recipe SET updated_at = now() WHERE …`) — but it produces a SQL expression that requires a DB round-trip to resolve, complicating unit tests that do not hit a real database. App-side UTC is the simpler and sufficient choice for MVP.

**Note on consistency with `created_at`:** `created_at` uses `server_default=text("now()")` which runs at INSERT time on the DB server. The slight discrepancy (server time for created, app time for updated) is acceptable in MVP. If clock consistency becomes critical, a future sprint can add `onupdate=text("now()")` to the `Recipe` model and migration.

*OQ3 resolved: ACCEPTED app-side UTC; rationale corrected per F3.*

## 7. Tests (`apps/api/tests/test_recipes_update.py`)

- Mock session: `AsyncMock` session with `MagicMock` result proxy

### V2 timestamp test strategy — fixed historical constant (F2)

To prevent the `test_update_recipe_updated_at_is_newer` test from flaking, the pre-update `updated_at` on the mock recipe row **must** be a fixed, clearly-historical constant — **not** `datetime.now()` at test-setup time. Mandate:

```python
_PAST = datetime(2026, 1, 1, 0, 0, 0, tzinfo=timezone.utc)
```

This is a module-level constant. The mock recipe is constructed with `row.updated_at = _PAST`. The handler calls `datetime.now(tz=timezone.utc)` which, on today's date (2026-06-02) and beyond, is always several months after `_PAST`. The assertion:

```python
from datetime import datetime, timezone

returned_updated_at = datetime.fromisoformat(response.json()["updated_at"])
assert returned_updated_at > _PAST
```

will **never flake** because `datetime.now(UTC)` in 2026 is always strictly greater than `datetime(2026, 1, 1, ...)`. This mirrors the `_NOW = datetime(2026, 6, 1, 12, 0, 0, tzinfo=timezone.utc)` pattern in `test_recipes_get.py`.

### Mock session shape for `update_recipe`

The handler calls `session.execute()` **twice** sequentially:
1. First call: load recipe → `result.scalar_one_or_none()`
2. Second call: exists check → `result.scalar_one()` (returns `True`/`False`)

The mock session must be configured with `side_effect=[result1_mock, result2_mock]` on `session.execute`.

After the dataset check passes, the handler calls `session.commit()` (AsyncMock), `session.refresh(recipe)` (AsyncMock — refreshes in place, no return value needed for mock), then constructs `RecipeOut.model_validate(recipe)`.

### Example mock session helper (pseudocode)

```python
_PAST = datetime(2026, 1, 1, 0, 0, 0, tzinfo=timezone.utc)


def _make_session_dep_for_update(
    recipe: MagicMock | None,
    dataset_exists: bool,
) -> Any:
    """Session override whose two execute() calls return recipe + exists bool."""
    async def _override() -> AsyncGenerator[AsyncMock, None]:
        session = AsyncMock()
        result1 = MagicMock()
        result1.scalar_one_or_none.return_value = recipe
        result2 = MagicMock()
        result2.scalar_one.return_value = dataset_exists   # True → 409, False → proceed
        session.execute = AsyncMock(side_effect=[result1, result2])
        session.commit = AsyncMock()
        session.refresh = AsyncMock()
        yield session
    return _override
```

For the 409 test, `dataset_exists=True`; for 200 tests, `dataset_exists=False`.

For the 404 test (recipe not found), only the first `execute()` call is reached, so `side_effect=[result1_none]` is sufficient (exists call never happens).

For the V2 timestamp test, the mock recipe must have `row.updated_at = _PAST` before the PUT executes.

## 11. Open Questions (all resolved)

**OQ1 — 409 vs 422 for locked recipe**
→ **ACCEPTED: 409 Conflict.** RFC 9110 §15.5.10 semantics match precisely: the request is well-formed; the server-side resource state (an existing `dataset` row) makes the mutation impossible. 422 implies the *input* is unprocessable, which is misleading here. Codebase precedent: F-037 already uses 409 for duplicate-name conflict.

**OQ2 — Should `description` be updatable?**
→ **ACCEPTED: yes, description is updatable.** `description` is a human-readable label with no transformation-contract semantics. It does not affect dataset reproducibility or lineage. The `model_fields_set` approach (F1) preserves the "omit = keep current; null = clear; string = update" behavior cleanly without a non-serializable sentinel.

**OQ3 — App-side `datetime.now(UTC)` vs `func.now()`**
→ **ACCEPTED: app-side UTC.** `func.now()` is technically valid in SQLAlchemy 2.0 and would render correctly in the UPDATE SET clause, but it produces a SQL expression that complicates unit tests that don't hit a real database. App-side `datetime.now(tz=timezone.utc)` is trivially mockable, round-trips as a concrete Python value, and is sufficient for MVP. See §6 for full rationale.

**OQ4 — Should `recipe_snapshot` on existing datasets be updated?**
→ **ACCEPTED: No.** `recipe_snapshot` is an immutable lineage record of what the recipe said at materialization time. Updating it retroactively would violate lineage invariant #1. Out of scope.

## 12. Out of Scope

- **Recipe versioning (new row instead of mutation):** The full version-management pattern (creating a successor recipe row, preserving the old row, linking datasets to specific versions) is a future feature. MVP is: "definition is freely editable until first materialization, then locked". Creating a new recipe row to supersede a locked one is user responsibility in MVP.
- **Partial/merge updates (PATCH):** PUT is a full replacement of `definition`. PATCH semantics (merge, JSONPatch, etc.) are deferred.
- **Name mutation:** `name` is not in `RecipeUpdate`. Renames require careful handling of the unique constraint and have no spec backing in F-040.
- **`schema_template_operator_id` mutation:** This FK field is excluded from `RecipeUpdate`; operator-linking semantics are undefined until the operator-linking sprint.
- **Authorization beyond owner-check:** Repository-level ACL (e.g., team-shared recipes) is deferred per CLAUDE.md MVP scope rules.

## 13. Risks and Mitigations

### Race condition: dataset INSERT between existence check and UPDATE commit

**Risk:** Between step 3 (exists check = False) and step 6 (commit), a concurrent worker could INSERT a `dataset` row pointing at this recipe (i.e., begin materializing it). The handler would proceed with the update, leaving the frozen snapshot diverged from the mutated definition.

**Likelihood:** Very low in practice. Dataset materialization is a multi-step async job (RQ worker) that takes seconds to minutes. The window between the existence check and the commit is sub-millisecond under normal load.

**MVP mitigation:** None (acknowledged). A correct future fix is one of:
- A DB trigger or `CHECK` constraint that rejects `recipe.definition` updates when `dataset.recipe_id` rows exist (but Postgres `CHECK` constraints cannot reference other tables directly — would need a trigger or a deferred FK variant).
- A pessimistic lock: `SELECT ... FOR UPDATE` on the `recipe` row followed by a re-check of dataset existence inside the same transaction (would need `SERIALIZABLE` isolation or explicit row lock).
- An application-level distributed lock (e.g. Redis) around the check-and-update pair.

These are deferred post-MVP. The race window is noted here for the implementer's awareness and for future sprint planning.

### Freeze-guard leak if `dataset.recipe_id` is ever null (F6)

**Observation:** The freeze check relies on `WHERE dataset.recipe_id = :id`. In the current ORM model, `recipe_id` is `nullable=True`. If a future code path materializes a Dataset row without setting `recipe_id` (partial write, bug, direct DB seeding, etc.), this freeze guard will silently fail to fire for that recipe. Out of scope for F-040; flag for the F-042 (dataset commit) reviewer to mandate `recipe_id IS NOT NULL` for every new dataset row and add a `NOT NULL` migration constraint if the design allows.

### `updated_at` clock skew

**Risk:** App server and Postgres server clocks could differ by a small amount, making `updated_at` (set app-side) inconsistent with `created_at` (set server-side).

**Mitigation:** Acceptable in MVP. Clocks on co-located services (Docker Compose, same VM) are typically synchronized via NTP to within microseconds. A future migration adding `onupdate=text("now()")` to the ORM model would eliminate this permanently.
