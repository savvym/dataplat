## 3. Schema Decisions and Clarifications

### 3.1 Primary key — BIGSERIAL

`BIGSERIAL PRIMARY KEY` maps to:
```python
id: Mapped[int] = mapped_column(sa.BigInteger, primary_key=True, autoincrement=True)
```
In the migration: `sa.Column("id", sa.BigInteger, sa.Identity(), primary_key=True)`.  
`sa.Identity()` is the SQLAlchemy 2.x canonical way to express `BIGSERIAL`; it generates `GENERATED BY DEFAULT AS IDENTITY` in PG10+ which is functionally equivalent.

### 3.2 JSONB columns

All `JSONB` columns (e.g. `source_metadata`, `output_schema`, `config_schema`, `default_config`, `definition`, `recipe_snapshot`, `stats`, `config`, `trigger_context`) use:
```python
sa.Column("source_metadata", postgresql.JSONB, server_default=sa.text("'{}'::jsonb"))
```
For columns where §4.1 specifies `DEFAULT '{}'`, the migration must include `server_default=sa.text("'{}'::jsonb")`. For nullable JSONB without a default (e.g. `output_schema`, `example_input`, `example_output`), no `server_default` is set and the column is `nullable=True`.

Implementer cross-check: the `operator` table must have exactly **19 columns** total (`id` + 18 from §4.1 lines 286–316). Count them before committing the migration.

### 3.3 TIMESTAMPTZ DEFAULT NOW()

```python
sa.Column("created_at", sa.DateTime(timezone=True), server_default=sa.text("now()"))
```
`server_default=sa.text("now()")` renders as `DEFAULT now()` in DDL. All timestamp columns with `DEFAULT NOW()` in §4.1 use this pattern. Nullable timestamps without a default (e.g. `started_at`, `ended_at` on the `run` table) are `nullable=True` with no `server_default`.

`document_variant.materialized_at` (§4.1 line 277) has `DEFAULT NOW()` in the spec and MUST include a `server_default`. The column is nullable (no `NOT NULL` in §4.1), so the correct declaration is:
```python
sa.Column("materialized_at", sa.DateTime(timezone=True), server_default=sa.text("now()"), nullable=True)
```
It must NOT be grouped with `started_at`/`ended_at` — those are genuinely default-free.

### 3.4 TEXT[] — PostgreSQL ARRAY

The `run` table has `asset_keys TEXT[] NOT NULL` and `partition_keys TEXT[] DEFAULT '{}'`. These use:
```python
sa.Column("asset_keys", postgresql.ARRAY(sa.Text), nullable=False)
sa.Column("partition_keys", postgresql.ARRAY(sa.Text), server_default=sa.text("'{}'"))
```
`asset_keys` carries `NOT NULL` explicitly in §4.1 so `nullable=False` is correct. `partition_keys` has no `NOT NULL` in §4.1 — only a `DEFAULT '{}'` — so `nullable=False` must NOT be added; omitting it leaves the column nullable, which matches the spec. `server_default=sa.text("'{}'")`  renders the Postgres empty-array literal correctly.

### 3.5 Partial unique index on document_variant

`CREATE UNIQUE INDEX idx_doc_canonical ON document_variant(source_id) WHERE is_canonical;`

In the migration:
```python
op.create_index(
    "idx_doc_canonical",
    "document_variant",
    ["source_id"],
    unique=True,
    postgresql_where=sa.text("is_canonical"),
)
```

### 3.6 ON DELETE CASCADE

The following FK relationships in §4.1 carry `ON DELETE CASCADE`:
- `source.collection_id` → `source_collection(id) ON DELETE CASCADE`
- `document_variant.source_id` → `source(id) ON DELETE CASCADE`

All other FK references (e.g. `source_collection.owner_id`, `recipe.owner_id`, `recipe.schema_template_operator_id`, `dataset.recipe_id`, `dataset.materialized_by`, `run.*`) carry no explicit `ON DELETE` clause in §4.1 — they default to `RESTRICT`. The migration must match §4.1 exactly: only the two listed FKs get `ondelete="CASCADE"`.

### 3.7 UNIQUE constraints in-line vs separate

§4.1 uses inline `UNIQUE` on:
- `users.email`
- `source_collection.name`
- `source.dagster_partition_key`
- `document_variant`: `UNIQUE (source_id, extractor_name, config_hash)` — composite, added as `sa.UniqueConstraint`
- `operator`: `UNIQUE (name, version)` — composite, added as `sa.UniqueConstraint`
- `recipe.name`
- `dataset`: `UNIQUE (recipe_id, version_tag)` — composite, added as `sa.UniqueConstraint`
- `run.dagster_run_id`

Single-column UNIQUE constraints are expressed as `unique=True` on the column. Multi-column UNIQUEs are expressed as `sa.UniqueConstraint(col1, col2)` appended to the `op.create_table` args.

### 3.8 downgrade() implementation

The `downgrade()` function drops tables in reverse FK dependency order to avoid FK violation errors:

```
run → dataset → recipe → operator → document_variant → source → source_collection → users
```

Each `op.drop_table()` call. Associated indexes are dropped automatically when their table is dropped in Postgres (no explicit `op.drop_index()` needed in `downgrade()` for indexes tied to those tables).

## 4. Alembic env.py Approach

`env.py` uses the standard SQLAlchemy 2.x async pattern. The key structure:

```python
import asyncio
import os
from sqlalchemy.ext.asyncio import create_async_engine
from alembic import context
from dataplat_api.db.models import Base

config = context.config
# Override sqlalchemy.url from DATABASE_URL env var:
config.set_main_option("sqlalchemy.url", os.environ["DATABASE_URL"])

def do_run_migrations(connection):
    context.configure(connection=connection, target_metadata=Base.metadata)
    with context.begin_transaction():
        context.run_migrations()

async def run_async_migrations():
    engine = create_async_engine(config.get_main_option("sqlalchemy.url"))
    async with engine.begin() as conn:
        await conn.run_sync(do_run_migrations)
    await engine.dispose()

def run_migrations_online():
    asyncio.run(run_async_migrations())

if context.is_offline_mode():
    raise RuntimeError("Offline mode is not supported — DATABASE_URL must be reachable")
else:
    run_migrations_online()
```

**`target_metadata` MUST be `Base.metadata`:** Since this sprint creates `models.py` with the `DeclarativeBase`, there is no reason to defer wiring `target_metadata`. The implementer MUST import `Base` from `dataplat_api.db.models` and pass `Base.metadata` so that future `alembic revision --autogenerate` works without re-editing `env.py`. `target_metadata=None` is not acceptable.

**Hard invariant compliance:** `create_async_engine` + `conn.run_sync(...)` is the only async-compatible pattern. Never use `engine_from_config` (that produces a sync engine). No `session.query()` anywhere in `env.py`.

## 5. Verification Plan

The `migration` layer in `verify/checks.sh` currently contains a minimal stub (lines 86–91). This sprint replaces it with the full implementation below. All commands are run from the repo root. The docker stack must already be up (`docker compose -f docker/docker-compose.dev.yml up -d`).

### Pre-flight — alembic installed in fastapi container

Before any migration check, confirm alembic is importable inside the container. If not, the container image needs a rebuild (see OQ-2):

```bash
docker compose -f docker/docker-compose.dev.yml exec -T fastapi uv run alembic --version
# Expected: "alembic 1.14.x"; exit code 0
# If this fails: docker compose build fastapi && docker compose up -d fastapi
```

This gates the rest of the migration layer — a `ModuleNotFoundError` at V1 is a build issue, not a migration bug.

### V1 — Migration command exits 0

**Criterion:** "Running the migration command exits 0 with no errors"

```bash
docker compose -f docker/docker-compose.dev.yml exec -T fastapi \
    uv run alembic upgrade head
# Expected: exit code 0, output includes "Running upgrade  -> 0001"
```

The command is run inside the `fastapi` container because that container has `alembic` installed and has `DATABASE_URL` set. Running it on the host would require a local `DATABASE_URL` pointing to `localhost:15432` and a local alembic installation — both fragile. The in-container path is authoritative.

### V2 — all 8 tables present in `information_schema`

**Criterion:** "`\dt` in psql lists all 8 expected tables"

Each table is verified with an exact-match query against `information_schema.tables` to eliminate substring false-positives (e.g. a `grep "source"` against `\dt` output would match `source_collection` even if `source` were absent):

```bash
for TABLE in users source_collection source document_variant operator recipe dataset run; do
  docker compose -f docker/docker-compose.dev.yml exec -T postgres \
    psql -U app -d platform -tAc \
    "SELECT 1 FROM information_schema.tables WHERE table_schema='public' AND table_name='$TABLE'" \
    | grep -q '^1$' \
    || { echo "FAIL: table '$TABLE' not found"; exit 1; }
  echo "  table $TABLE: OK"
done
# Expected: 8 "OK" lines; exit code 0
```

This approach is immune to substring collisions (`source` vs `source_collection`) and names the missing table explicitly on failure.

### V3 — Idempotent re-run

**Criterion:** "Re-running migration is idempotent (exits 0, no duplicate table errors)"

```bash
docker compose -f docker/docker-compose.dev.yml exec -T fastapi \
    uv run alembic upgrade head
# Expected: exit code 0, output is "INFO  [alembic.runtime.migration] Running upgrade ... -> 0001_baseline_schema"
# On second run the output is just the version check with no "Running upgrade" line — that is correct Alembic behaviour
```

The check passes if the command exits 0. No `duplicate table` or `already exists` error in stderr.

### V4-extra — Round-trip downgrade then upgrade (proves downgrade correctness)

```bash
docker compose -f docker/docker-compose.dev.yml exec -T fastapi \
    uv run alembic downgrade base
docker compose -f docker/docker-compose.dev.yml exec -T fastapi \
    uv run alembic upgrade head
# Expected: both commands exit 0
```

This extra check is included in `verify/checks.sh migration` to ensure the `downgrade()` function is correct. It is NOT listed in `feature_list.json` V1–V3 but is listed as a best-practice check in the SKILL.md. If it fails, the migration is considered incomplete.

### Full updated `migration` block in `checks.sh`

```bash
migration)
  exists apps/api/alembic || { echo "no alembic yet"; exit 0; }
  COMPOSE="docker/docker-compose.dev.yml"
  API="docker compose -f $COMPOSE exec -T fastapi"
  PG="docker compose -f $COMPOSE exec -T postgres"

  echo "--- migration pre-flight: alembic installed in fastapi container ---"
  $API uv run alembic --version \
    || { echo "FAIL: alembic not in fastapi container — run: docker compose build fastapi && up -d fastapi"; exit 1; }

  echo "--- migration V1: upgrade head exits 0 ---"
  run "$API uv run alembic upgrade head"

  echo "--- migration V2: all 8 tables present ---"
  for TABLE in users source_collection source document_variant operator recipe dataset run; do
    $PG psql -U app -d platform -tAc \
      "SELECT 1 FROM information_schema.tables WHERE table_schema='public' AND table_name='$TABLE'" \
      | grep -q '^1$' \
      || { echo "FAIL: table '$TABLE' not found"; exit 1; }
    echo "  table $TABLE: OK"
  done

  echo "--- migration V3: idempotent re-run ---"
  run "$API uv run alembic upgrade head"

  echo "--- migration V4-extra: downgrade base + upgrade head round-trip ---"
  run "$API uv run alembic downgrade base"
  run "$API uv run alembic upgrade head"
  ;;
```

### OQ-2 — Alembic installation and container rebuild

`alembic==1.14.1` is added to `apps/api/pyproject.toml`. The `fastapi` service bind-mounts `apps/api/` at `/app` for live reload, but `uv sync` is only run at image build time (in `docker/api/Dockerfile`). This means:

**After adding alembic to `pyproject.toml`, the implementer MUST rebuild the `fastapi` image:**
```bash
docker compose -f docker/docker-compose.dev.yml build fastapi
docker compose -f docker/docker-compose.dev.yml up -d fastapi
```
Without this, `uv run alembic` inside the container will fail with `No module named alembic`.

The `docker/api/Dockerfile` likely uses `uv sync` or `pip install -e .` — the implementer must verify which pattern is used and confirm alembic is installed after the rebuild. If the Dockerfile uses `pip install` directly (not pyproject.toml), the Dockerfile must also be updated.

### OQ-3 — Alembic version pin

`alembic==1.14.1` is proposed. This is the latest stable release as of the sprint date. It supports SQLAlchemy 2.0+ and the async engine pattern. The implementer should verify this tag is available on PyPI; if not, use the most recent available 1.14.x and note the substitution in a comment.

### OQ-4 — `alembic.ini` `sqlalchemy.url` placeholder

`alembic.ini` must have a `sqlalchemy.url` entry (Alembic requires the key to exist even if overridden at runtime). The placeholder value should be `postgresql+asyncpg://placeholder/placeholder` with a comment explaining it is always overridden by `env.py` via `DATABASE_URL`. This prevents accidental connection to the placeholder DSN if `env.py` is misconfigured.

### OQ-5 — `config.py` vs existing app structure

There is currently no `apps/api/dataplat_api/config.py`. The `fastapi` service has `DATABASE_URL` in its env. The new `config.py` will use `pydantic-settings` (`BaseSettings`) to read `DATABASE_URL`. If `pydantic-settings` is not yet in `pyproject.toml`, it must be added (`pydantic-settings==2.x`). The implementer should check whether the existing `main.py` uses any settings object; if not, `config.py` is a fresh creation.

### OQ-6 — `uv.lock` update

Adding `alembic` and potentially `pydantic-settings` to `pyproject.toml` will require regenerating `apps/api/uv.lock`. The implementer must run `cd apps/api && uv lock` after editing `pyproject.toml` and commit the updated lockfile alongside the other changes.
